## 작업 내용

앨범 상세 페이지에서 MP3 트랙 플레이 정상 작동 여부 점검 및 버그 수정.

## 진단 결과

### AudioPlayer 컴포넌트 (`frontend/src/components/audio/AudioPlayer.tsx`)
- `<audio src={file_url}>` 태그로 재생
- `isDisabled = !src || status !== 'completed'` 조건으로 비활성화 제어
- 컴포넌트 자체 로직은 정상

### 핵심 버그 발견

`_download_and_save_track()` 함수가 트랙 파일을 로컬에 저장한 후 DB `file_url`을 **컨테이너 내부 경로** (`/app/uploads/tracks/{id}.mp3`)로 저장하고 있었음.

- 브라우저는 컨테이너 내부 경로에 직접 접근 불가 → AudioPlayer 재생 불가
- 올바른 형태: `http://localhost:8001/api/v1/files/local/tracks/{id}.mp3`

## 변경된 파일

### `backend/app/api/music.py`
- `from app.core.config import settings` import 추가
- `_download_and_save_track()` 내부에서 `file_url`을 HTTP URL로 저장하도록 수정:
  ```python
  # 수정 전
  track.file_url = local_path  # /app/uploads/tracks/{id}.mp3
  # 수정 후
  http_url = f"{settings.FILE_BASE_URL}/api/v1/files/local/tracks/{track_id}.mp3"
  track.file_url = http_url    # http://localhost:8001/api/v1/files/local/tracks/{id}.mp3
  ```

### `backend/app/api/albums.py`
- `_fetch_bytes()` 함수 최적화:
  - 로컬 HTTP URL (`{FILE_BASE_URL}/api/v1/files/local/...`) 감지 시 파일 경로로 직접 변환해 읽음 (httpx 루프백 요청 불필요)
  - 케이스 정리: `/app/uploads` 직접 경로 / 로컬 HTTP URL / 외부 URL 3가지로 명확히 분리
