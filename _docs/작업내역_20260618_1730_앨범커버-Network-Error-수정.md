## 작업 내용

"생성된 커버 가져오기 → 앨범에 적용" 시 발생하는 `AxiosError: Network Error` 수정.

## 근본 원인

커버 생성 서비스(GPT Image 1, PIL fallback)가 이미지를 파일에 저장하지 않고 `data:image/png;base64,...` 형식(수백KB~수MB)으로 DB에 저장하고 있었음. 이걸 `PATCH /albums/{id}` 요청 body에 그대로 담아 전송 → 브라우저 Network Error.

## 변경된 파일

### `backend/app/api/files.py`
- `GET /local/{filename}` → `GET /local/{filename:path}` 로 변경
  (서브디렉토리 `covers/` 경로 지원)

### `backend/app/api/albums.py`
- 상단에 `base64`, `os`, `Cover`, `ApplyCoverRequest` 추가
- `POST /{album_id}/cover` 신규 엔드포인트 추가:
  - `cover_id` 수신 → Cover DB 조회
  - base64 이미지 → 디코딩 → `/app/uploads/covers/{filename}.png` 파일 저장
  - `cover.image_url` 을 실제 파일 URL(`BACKEND_CALLBACK_URL/api/v1/files/local/covers/...`)로 업데이트
  - `album.cover_url` = 해당 URL 저장

### `backend/app/services/cover.py`
- `_save_cover_to_file(cover_id, size, image_url)` 헬퍼 함수 추가
- `_process_cover()`: 앞으로 생성되는 커버는 base64가 아닌 실제 파일 URL로 저장

### `frontend/src/app/(dashboard)/albums/[id]/page.tsx`
- `useQueryClient` import 추가
- `selectedCoverId`, `isApplyingCover` 상태 추가
- `CoverGroup.images` 타입에 `cover_id: string` 추가
- `groupCompletedCovers()`: 각 이미지에 `cover_id` 포함
- 이미지 클릭 시 `selectedCoverId`도 함께 저장
- `handleApplyCover()`: `PATCH` 대신 `POST /albums/{id}/cover { cover_id }` 호출
  (base64 → 파일 변환은 백엔드에서 처리)
- 닫기 핸들러에 `setSelectedCoverId(null)` 추가
- "앨범에 적용" 버튼: `isApplyingCover` 로딩 상태 사용

## 동작 흐름 (수정 후)

1. 다이얼로그에서 커버 클릭 → `selectedCoverUrl` (미리보기 링) + `selectedCoverId` (API용)
2. "앨범에 적용" 클릭 → `POST /albums/{id}/cover { cover_id }`
3. 백엔드: base64 디코딩 → 파일 저장 → `cover.image_url` = 실제 URL → `album.cover_url` = 해당 URL
4. 프론트: 쿼리 무효화 → `useAlbumById` 리페치 → 헤더 커버 즉시 갱신
