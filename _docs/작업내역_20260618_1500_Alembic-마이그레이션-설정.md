DB 볼륨을 날리지 않고 스키마 변경할 수 있도록 Alembic 마이그레이션을 설정했습니다.

## 변경된 파일

- `backend/alembic/env.py`: asyncpg 드라이버를 올바르게 사용하도록 URL 처리 수정
- `backend/alembic/script.py.mako`: 마이그레이션 파일 생성 템플릿 추가 (누락됐던 파일)
- `backend/alembic/versions/91ca4bf006d8_initial.py`: 초기 베이스라인 마이그레이션 (현재 스키마를 기준점으로 기록)
- `backend/app/core/database.py`: 시작 시 `create_all` 대신 `alembic upgrade head` 실행으로 변경
- `migrate.ps1`: 마이그레이션 워크플로 도우미 PowerShell 스크립트

## 앞으로의 워크플로

### 모델 변경 시 (예: 컬럼 추가)

```powershell
# 1. 모델 파일 수정 (예: backend/app/models/vocal.py 에 컬럼 추가)

# 2. 마이그레이션 파일 생성 (컨테이너 안에서 실행 후 호스트로 자동 복사)
.\migrate.ps1 new "vocal에 pitch 컬럼 추가"

# 3. 백엔드 재빌드 (마이그레이션 파일이 이미지에 포함됨)
docker-compose build --no-cache backend
docker-compose up -d backend

# → 시작 시 alembic upgrade head 자동 실행 → 새 컬럼만 ALTER TABLE로 추가
# → 기존 데이터 보존!
```

### 기타 명령

```powershell
.\migrate.ps1 current    # 현재 적용된 마이그레이션 버전
.\migrate.ps1 history    # 전체 마이그레이션 이력
.\migrate.ps1 up         # 수동으로 upgrade head 실행
```

## 주요 결정 사항

- **폴백 전략**: alembic 실패 시 `create_all`로 자동 폴백 (완전 새 DB에서도 안전)
- **마이그레이션 파일**: 컨테이너에서 생성 → 호스트로 복사 → 이미지에 bake-in
- Windows 경로 공백 문제로 볼륨 마운트 방식 불가 → 빌드 시 포함 방식 채택
