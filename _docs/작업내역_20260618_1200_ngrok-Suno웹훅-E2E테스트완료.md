## 발견된 문제 및 수정

### 웹훅 핸들러 구조 불일치

**문제**: sunoapi.org 실제 콜백 구조가 예상과 달랐음

```json
// 실제 콜백 구조
{
  "code": 200,
  "msg": "Text generated successfully.",
  "data": {
    "task_id": "f11425cdb6a507bc6e003421cb4f0f92",   // 루트가 아닌 data 안에 있음
    "callbackType": "text",                            // text / audio / first_audio
    "data": [
      {
        "audio_url": "",                               // text 콜백 시 비어있음
        "stream_audio_url": "https://musicfile.removeai.ai/...",  // 실제 오디오
        "source_stream_audio_url": "https://audiopipe.suno.ai/...",
        "image_url": "https://...",
        "title": "Sunlit Detour",
        "tags": "pop, happy"
      }
    ]
  }
}
```

**수정 내용** (`backend/app/api/music.py`):
- `task_id` 추출: `body.get("taskId")` → `body["data"]["task_id"]`
- `audio_url` 폴백: `audio_url` 비어있으면 `stream_audio_url` 사용
- `callbackType` 분기 처리:
  - `text`: 가사 생성 완료, `stream_audio_url`로 completed 처리
  - `audio` / `first_audio`: 최종 오디오 완료
- track 미발견 시 `200` 반환 (sunoapi.org 재전송 방지)

### `backend/app/api/music.py`
- `POST /api/v1/music/webhook/suno` 핸들러 로직 전면 수정
  - `callbackType` 기반 분기 처리
  - `data.task_id` 경로로 task_id 추출
  - `stream_audio_url` 폴백 추가

## 전체 흐름 (확정)

```
프론트엔드
  → POST /api/v1/music/generate
  → Track 생성 (status=processing)
  → 백그라운드: sunoapi.org API 호출
      callBackUrl = https://ngrok주소/api/v1/music/webhook/suno
  → taskId → Track.task_id 저장

sunoapi.org (약 30초 후)
  → POST https://ngrok주소/api/v1/music/webhook/suno
      {callbackType: "text", data.task_id: "...", data.data[0].stream_audio_url: "..."}

백엔드 웹훅 핸들러
  → task_id로 Track 조회
  → Track.status = "completed"
  → Track.file_url = stream_audio_url 저장

프론트엔드 폴링
  → GET /api/v1/music/{track_id}/status
  → status=completed, file_url 수신 → 재생 가능
```
